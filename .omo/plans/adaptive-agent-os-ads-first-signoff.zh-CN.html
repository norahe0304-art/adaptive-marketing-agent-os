Strategy Blueprint

我们在做 Adaptive Agent OS。Ads 是第一条真实闭环，不是最终边界。

核心目标不是再堆几个 agent，而是把一个混乱行业的运营图谱建出来，让 agent 在图谱上按角色、证据、权限和学习回写去跑。 Jetpartner 是 v0 试点，30x Ads 是第一台确定性 domain engine。

Strategy Lock

这版蓝图先锁战略取舍。通用 OS 可以扩展到 SEO、Content、Lifecycle，但 v0 必须用 Ads/Jetpartner 证明一条真实运营闭环。

OS first先设计 Adaptive Agent OS 的通用骨架，再把 Ads 当第一个 vertical mount。
Ads first广告账户有数据、权限、报告、proposal、approval gate，最容易证明闭环。
Jetpartner first它有真实 tenant truth：qualified lead 优先、private aviation intent、人工审批边界。
Read-only firstv0 只做观察、诊断、proposal 和 evidence，不自动改 Google Ads。
Evidence first每个判断都必须能回到 artifact、report、query、memory 或 workflow record。
Delta firstsession 先沉淀 semantic delta。稳定后才升级为 skill candidate。

三层设计，以及每层为什么存在

三层不是 L1-L6 的大而全层级，而是把系统复杂度切成三个不会互相污染的面：Truth 决定相信什么，Work 决定怎么做，Learning 决定怎么变聪明。

Layer 1

Truth Layer - 运营图谱和事实边界

把 tenant truth、industry truth、role memory 从 base agent 里拆出去。

What
Jetpartner truth、private aviation intent、qualified lead 标准、权限边界。
Why
如果把所有客户事实塞进 agent prompt，OS 会膨胀、冲突、不可 scale。
Owns
tenant overlay、industry overlay、approved examples、permissions。
Not
不保存通用 agent 指令，不把 Jetpartner 写死进 base agent。
Layer 2

Work Layer - 角色、workflow、engine、验证

让 agent 不靠聊天临场发挥，而是按 typed workflow 和 execution governance 工作。

What
agent-role package、capability registry、Flyte-style DAG、30x Ads、OMO loop。
Why
Agent 负责判断和编排，domain engine 负责确定性执行，OMO 负责证据和 review gate。
Owns
plan、delegate、observe、proposal、approval gate、readback、QA。
Not
不让 prompt 直接 mutate 账户，不让 OMO 变成广告 API 引擎。
Layer 3

Learning Layer - session delta 到 skill factory

把每次运营交互沉淀为可审计学习，不让每个 session 都变成新 skill。

What
GEB semantic delta、memory patch、skill patch、session distillation、optional Record & Replay。
Why
大多数学习发生在 session 内部，先分类写回，稳定可复现才进入 skill candidate。
Owns
tenant-only delta、industry playbook、workflow patch、new skill candidate、protocol update。
Not
不自动安装未经验证的新 skill，不把学习垃圾写进全局协议。

业务逻辑：Jetpartner v0 怎么跑

v0 只验证一个真实动作：Jetpartner Ads Read-only Review Loop。它输出 proposal、evidence、approval gate、semantic delta 和 skill candidate decision。

01User asks“看看最近有没有垃圾流量，qualified lead 为什么不行。”
02Resolve Truth读取 Jetpartner memory、permissions、qualified lead rules。
03Observe调用 30x Ads/read-only artifacts，不直接改账户。
04Classify按 private aviation intent、search term、lead quality 归因。
05Draft Proposal生成 negative keyword、budget、landing 或 tracking 建议。
06Review Gate列 evidence、risk、dry-run plan，等待人类批准。
07Learn写 semantic delta，判断是否只是 memory patch 或 skill candidate。
这不是 dashboard，也不是 auto-apply agent。第一条闭环必须先证明：agent 能看懂业务真值，能安全提出操作方案，能留下证据，能把经验写回。

Agent 逻辑：谁做什么，怎么协作

OMO 里单个 agent 好用，不是因为角色名字多，而是因为它把身份、边界、工具、证据、review gate 和完成标准合成一个可执行角色。

运行时 orchestrator

Base agent 接到任务后不直接输出答案，而是先解析 tenant、选择 role package、生成 typed workflow，再把能力路由给正确工具和 reviewer。

  • Resolve - 找 tenant truth、industry truth、permissions。
  • Select - 选 ads-adaptive-agent，加 Jetpartner overlay。
  • Build - 生成 search-term-review / lead-quality-review DAG。
  • Route - 30x Ads 执行观察，roles 做判断，GEB 做回写。
  • Verify - OMO evidence + QA + review gate。

协作角色组

这些 role 不是“多开几个聊天机器人”，而是把运营任务拆成可审计职责。每个 role 都有输入、输出、权限和失败标准。

  • Explorer - 查本地事实、repo、artifact、历史报告。
  • Librarian - 查外部规则、平台文档、行业依据。
  • Metis - 找风险、冲突、隐藏假设和反例。
  • Plan - 写可执行 workflow 和验收标准。
  • Momus - 挑计划毛病，阻止幻觉执行。
  • Executor / QA - 执行 read-only steps，做 evidence readback。

Business Agent

  • ads-adaptive-operator - 通用 Ads operator。
  • jetpartners-ads-operator - tenant overlay，只持有 Jetpartner truth。
  • private-aviation-intent-judge - 行业 intent playbook，可复用到类似客户。

Execution Agents

  • explorer / librarian - evidence collection。
  • plan / executor - workflow construction and run。
  • momus / QA gate - review, readback, contradiction check。

Learning Agents

  • session-distiller - 从对话里抽取 learning delta。
  • GEB keeper - 决定写到 L1/L2/L3、tenant memory 或 industry playbook。
  • skill-curator - 只有稳定流程才产出 skill candidate。

Role 怎么设计

agent-role 的价值是把“谁在做事”变成 package boundary：身份、记忆、权限、工具、MCP/capability 边界、生命周期、非目标。OMO 的价值是把这个 role 变成能完成事的执行协议。

Role Package Schema

  • identity这个 role 是谁，服务哪个工作面。
  • purpose它存在的唯一目标是什么。
  • when_to_use什么时候被 orchestrator 选中。
  • inputs需要哪些 truth、artifact、tools。
  • capability_surface允许调用哪些能力：browser、repo、ads engine、CRM、docs。
  • mcp_boundary每个 MCP/tool 的 read/write/dry-run/apply 权限。
  • outputs必须产出什么 artifact。
  • approval_policy哪些动作必须 human approval，哪些只能 proposal。
  • evidence_contract每次工具调用必须留下什么 readback 和 evidence。
  • memory_scope哪些学习能写回哪里。
  • non_goals明确拒绝做什么，避免越权。
  • success_criteria怎样算完成，怎样算失败。
  • learning_rulessession delta 如何分类沉淀。

示例：jetpartners-ads-adaptive-operator

这个 role 不应该是一个巨大的 prompt。它应该是 base Ads operator 加 Jetpartner overlay：通用广告技能在 base，客户真值在 overlay。

  • Identity - Jetpartner 的 Ads 运营审查员，不是全行业营销顾问。
  • Inputs - Jetpartner memory、30x Ads artifacts、qualified lead truth、permissions。
  • MCP Boundary - 30x Ads 只读 observe；Google Ads mutation 只能 dry-run/proposal；CRM/tenant memory 默认 read-only。
  • Output - evidence-backed proposal，而不是直接改广告账户。
  • Approval - mutation 必须 human approval，v0 默认 read-only。
  • Learning - tenant-only 写 Jetpartner memory，通用 private aviation 才进 industry playbook。
  • Boundary - 不把 SEO、CRM、Content 全塞进这个 role。

Capability / MCP Boundary

  • Read - 可以读 repo、report、docs、tenant memory、approved examples。
  • Dry-run - 可以生成 Google Ads proposal、diff、rollback plan。
  • Apply - 默认禁止；必须 human approval + OMO review gate。

通用协议字段

  • server - MCP/tool 名称，例如 30x-ads、google-ads、browser、docs。
  • scope - tenant、industry、global 三种作用域，不能越界。
  • mode - read / observe / dry_run / propose / apply。

Why It Matters

  • 安全 - 没有 MCP 边界，agent 会把“能看”误认为“能改”。
  • 复用 - 同一个 Ads role 可以挂不同 tenant overlay 和 tool surface。
  • 审计 - 每次 tool use 都能回到 evidence、approval、semantic delta。
