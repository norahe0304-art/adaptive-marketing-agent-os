一、Shared Adaptive Marketing Agent OS Protocol

这是底层协议正文。它只定义所有 marketing agent 必须遵守的结构，不负责 Ads/Event 具体业务。

Boundary

协议层负责什么

协议层只定义所有 marketing agent 共同遵守的“操作系统接口”。它回答：一个 agent 长什么样、能用什么工具、什么时候必须停下来、做完后怎么留下证据和学习回写。

  • Role schema：规定每个 role package 必须有 identity、instructions、skills、memory_scope、tools、host_adapters、lifecycle、approval_policy、evidence_contract。通过标准：Ads/Event/SEO 都能用同一份 schema 描述自己。
  • Capability / MCP boundary：规定 base role 默认只到 read、observe、dry_run、propose；apply 只能出现在 workflow-scoped apply lab，并且必须有 runtime binding、typed approval、evidence、exact scope 和 readback。通过标准：工具权限可以审计，domain agent 不能偷偷拿到全局工具。
  • Approval / evidence model：规定哪些动作需要 human approval，approval receipt 和 evidence packet 必须包含什么。通过标准：任何高风险动作都能追到人、时间、对象、证据和 readback。
  • OMO governance：规定 agent 做事必须经过 trigger、self-check、plan、delegate、evidence、QA、review gate、readback。通过标准：agent 不是聊天发挥，而是按执行协议完成工作。
  • GEB delta：规定结果如何进入 tenant memory、playbook、workflow patch、skill candidate 或 protocol update，同时维护 L1/L2/L3 文档同构。通过标准：系统不会失忆，也不会把客户事实污染到 base role。
Boundary

协议层不负责什么

协议层不是超级 agent，也不是业务 runtime。它不能替 Ads、Event、SEO 做领域判断；它只给这些 agent 一个共同的设计、执行和审计框架。

  • 不实现 Ads role：Ads search term、lead quality、campaign optimization 属于 JP Ads agent。
  • 不实现 Event role：HubSpot pages、emails、workflows、event launch kit 属于 Caylent Event agent。
  • 不写 platform runtime：Google Ads、HubSpot、Salesforce、Slack adapter 的真实 API/runtime 不在协议层实现。
  • 不把 tenant adapter 当 OS core：具体 Slack adapter 只能由 tenant overlay 选择，不能进入 shared/base 协议。
  • 不存 tenant truth：JP qualified lead、Caylent approval owner、客户 portal id 等事实只能放 tenant overlay 或 source pointer。
Scale Rule

新 agent 如何接入

新增 agent 不是重新设计一套协议，而是填写同一张接入表。只要这张表填得完整，新 agent 就能被执行、审计和进化。

  • 1. 声明 domain：Ads、Event、SEO、Content、Lifecycle、Partner Ops。例子：JP 是 Ads；Caylent 是 Event；未来 SEO agent 也是同一入口。
  • 2. 声明 base role：描述通用领域能力，不含客户事实。例子：ads-adaptive-operator 不知道 Jetpartner；event-adaptive-operator 不知道 Caylent。
  • 3. 声明 tenant overlay：只放稳定 operating contract 和 source pointers。例子：Jetpartner qualified lead rule；Caylent approval owners。
  • 4. 声明 workflow contract：写清 inputs、outputs、task graph、evidence、approval gate、failure behavior、readback。例子：JP readonly review；Caylent launch workflow。
  • 5. 声明 learning route：结果进入 tenant memory、industry playbook、workflow patch、skill candidate 或 protocol update。例子：重复稳定的 session 才能升级成 skill candidate。

二、执行门禁：Capability、Evidence、OMO、GEB

这部分决定 agent 能不能安全做事。核心是：可观察、可提案、可审计，但 v1 不默认 live mutation。

Apply Gate

Capability modes

  • read：读取已有事实和文档。
  • observe：观察外部状态，不改变系统。
  • dry_run：模拟执行，生成风险和预期结果。
  • propose：输出建议和 evidence packet。
  • apply：V1 可试，但只能在 workflow-scoped apply lab 里执行。

Apply lab 必须同时具备 runtime binding、risk class、tenant/account binding、named human approver、approval receipt、pre-apply evidence、post-apply readback、rollback 或 irreversible acknowledgement。缺一项就停在 propose。

Evidence

Evidence Gates

  • request / brief evidence
  • source evidence
  • plan / proposal evidence
  • approval evidence when risk requires it
  • final readback evidence
  • failure evidence for blocked or aborted actions

Evidence record fields

evidence_id, source_url, created_at, actor, tenant, account_or_campaign, before_state, after_state, approval_receipt, readback_summary.

OMO

Execution Governance

trigger
  -> self-check
  -> plan
  -> delegate
  -> gather evidence
  -> execute by permission
  -> QA
  -> Metis/Momus review gate
  -> readback
GEB

GEB Semantic and Structural Delta

GEB 不是单纯 learning。每次结果都要决定 semantic placement 和 structural documentation impact。

  • Learning delta：tenant memory、industry playbook、role patch、workflow patch、skill candidate、protocol update。
  • Structural delta：任何 create / move / rename / responsibility shift 都要维护 L1/L2/L3 同构。
  • 固定原则：map and terrain must stay aligned。

三、Seed Agent 1：JP Ads Adaptive Agent

这是第一个落地 agent。它消费 shared protocol，负责 Ads domain 和 Jetpartner tenant overlay，不定义 OS protocol。

Base Role

ads-adaptive-operator

  • 操作广告账户的 read-first evidence loop。
  • 支持 Google Ads、Meta、LinkedIn、TikTok、Microsoft、Apple、YouTube 等能力面。
  • 分析 campaign structure、search terms、lead quality、budgets、creatives、landing pages、conversion quality。
  • 必须先产生 proposal；apply 只能通过 workflow apply lab 试验。
  • 不得包含 Jetpartner、Eric、Benton、reports URL、Contacted、Pre-Qualified、private aviation bad intent。
Host

Ads host adapters

host_adapters:
  required: []
  optional:
    - codex
    - portal
    - slack
  preferred: {}
  unsupported: []
  notes: "Ads role can run through optional hosts without requiring one in the base role."
Overlay

jetpartners-ads-operator

  • mounts_on：ads-adaptive-operator
  • tenant memory boundary
  • qualified lead truth
  • Supabase qualified lead source
  • report surface
  • known bad intent such as private aviation mismatch
  • read-only default
  • human approval for mutations

Jetpartner workflow

01Request确认 tenant 和 account。
02Gather采集 ads/search-term/lead evidence。
03Classify判断 lead quality。
04Propose提出 negatives、budget、copy、landing changes。
05Approve等待 human approval receipt。
06Apply Labapproved low-risk reversible operation。

JP Ads sign-off rules

  • Base Ads role 必须 tenant-neutral。
  • Jetpartner truth 只能在 overlay。
  • Workflow 默认 proposal；apply lab 只跑 approved low-risk reversible operations。
  • Approval receipt 必须包含 human approver、approved action、tenant/account、timestamp、evidence packet、exact mutation scope。
  • Overlay tenant fact 必须包含 source_of_truth、evidence_url、owner、last_verified_at、review_after、promotion_target 或 expiry_reason。

四、Seed Agent 2：Caylent Event Adaptive Agent

这是第二个落地 agent。它证明协议不是 Ads-only。Base Event 保持 host-neutral，tenant overlay 才绑定具体协作入口。

Base Role

event-adaptive-operator

  • 运行 B2B event launch workflows。
  • 协调 event pages、emails、workflows、lists、campaign associations、approval packets。
  • 创建 launch assets 前必须读取 CRM/account context。
  • publish/send/activate 前必须先 draft/propose。
  • 不得包含 Caylent tenant-only facts、real HubSpot portal IDs、real Salesforce account IDs。
Base Host

Event host-neutral

host_adapters:
  required: []
  optional:
    - portal
    - codex
    - slack
  preferred: {}
  unsupported: []
  notes: "Base Event role is host-neutral; tenant overlays may require a collaboration host."
Caylent Overlay

caylent-event-operator

  • mounts_on：event-adaptive-operator
  • Caylent tenant memory boundary
  • Caylent event naming conventions
  • Caylent HubSpot portal binding notes
  • Caylent Salesforce read-only context notes
  • Caylent approval owners and receipts
  • required Slack host with overlay-defined adapter

Caylent host adapter instance

host_adapters:
  required:
    - slack
  optional:
    - portal
    - codex
  preferred:
    slack: overlay-defined
  unsupported: []
  notes: "Caylent Event uses Slack as the primary collaboration host; the preferred adapter is defined in the overlay."

Capability surface

  • hubspot.pages：draft/propose，publish requires approval。
  • hubspot.emails：draft/propose，send requires approval。
  • hubspot.workflows：draft/propose，activate requires approval。
  • hubspot.lists：read/propose，list updates require approval。
  • salesforce：read only。
  • docs：draft briefs、launch plans、approval packets。

Caylent workflow

01Briefnormalize Caylent event brief。
02Contextread Salesforce/account context。
03Draftdraft HubSpot page/email/workflow/list assets。
04Campaignassociate campaign id。
05Approvalprepare Slack approval packet and wait receipt。
06Readbackevidence readback and semantic delta handoff。

Caylent sign-off rules

  • Base Event role 必须 tenant-neutral 和 host-neutral。
  • 具体 Slack adapter 只能在 tenant overlay required/preferred。
  • Default stop point 是 approval packet 和 draft assets。
  • Publish/send/activate/list updates 必须 explicit approval。
  • Evidence contract 必须有 event brief、campaign id、asset urls、CRM/account context source、approval receipts、owner identity、launch timestamp、failure/readback notes。
